Using The API

During the creation of a Qeo reader or writer you can optionally provide a PolicyUpdateListener object which has an on policy update callback method. If the callback is specified then Qeo will call it:

  1. When creating the reader or writer,
  2. Whenever a Qeo Policy update is received,
  3. When requesting a Qeo Policy re-evaluation (see below).

When the callback method gets called:

  1. It will receive a Qeo Policy identity.
  2. It should decide whether it will allow or deny the provided identity from being read from or written to.

The callback will be called:

  1. Once for each user (identity) to whom you're allowed to communicate in your realm. This is defined by the policy of the realm.
  2. Once with a null identity to signal that the end of the Qeo Policy was reached. 

In some cases an application may want to reconsider a decision made earlier on whom to allow or deny for a certain reader or writer.  For this reason each entity also has a policy update method that allows the application to request a re-evaluation of the Qeo Policy.  When calling this method the on policy update callback will be called again for each relevant identity in the Qeo Policy.

The PolicyUpdateListener.onPolicyUpdate Callback

The callback only receives a single arguments:

  • A Qeo Policy identity,

The callback should return one off:

  • AccessRule.ALLOW to allow the provided identity from being read from or written to.
  • AccessRule.DENY to deny the provided identity from being read from or written to.
Example
@Override
public AccessRule onPolicyUpdate(Identity identity)
{
    return AccessRule.DENY; //deny all users
}

It will be called multiple times. Once for every user in the realm and a last time with identity set to null to indicate all the updates have been done.

Registration of the listener during reader/writer creation

The PolicyUpdateListener is passed as an argument to the reader/writer creation call:

mQeo.createEventWriter(ChatMessage.class, new PolicyUpdateListener() {
	//implement the required methods
});

Note: the reader/writer creation will block untill all the onPolicyUpdate callbacks for that reader/writer are handled. This ensures the reader/writer can only be used after the correct policies are set.

Forcing a policy update

If you want to change a policy, you need to request an update in order for the onPolicyUpdate callback to be called again

//trigger an update to the event writer
mWriter.updatePolicy();
